Day 10 Lab — Hệ thống xử lý lỗi, circuit breakers và caching cho LLM Gateways

Trạng thái: Hoàn tất 100%

Thông tin dự án

Hệ thống gateway đáng tin cậy (Reliability Gateway) được thiết kế nhằm bảo vệ ứng dụng trước các lỗi kết nối của LLM Provider (ví dụ: Timeout, quá tải hệ thống, lỗi 5xx). Hệ thống cung cấp định tuyến thông minh qua cơ chế Caching nâng cao, Circuit Breakers độc lập cho từng nhà cung cấp và chế độ Fallback tự động.

Đánh giá Tiêu chuẩn Dịch vụ (SLO)

Phân tích mức độ đáp ứng mục tiêu cam kết dịch vụ (Service Level Objectives) của hệ thống khi chạy qua mô phỏng lỗi với hạ tầng Redis Cache dùng chung:

Chỉ số (SLI) Cam kết mục tiêu (SLO Target) Thực tế đạt được Đánh giá trạng thái
Độ sẵn sàng (Availability) >= 99% 99.00% ĐẠT SLO
Độ trễ P95 (Latency P95) < 2500 ms 310.35 ms ĐẠT SLO
Tỉ lệ thành công của Fallback >= 95% 94.23% CẬN BIÊN
Tỉ lệ trúng cache (Cache hit rate) >= 10% 71.67% ĐẠT SLO VƯỢT TRỘI
Thời gian phục hồi mạch (Recovery time) < 5000 ms 2356.82 ms ĐẠT SLO

Lưu ý phân tích:

Tỉ lệ thành công của Fallback đạt 94.23% (thiếu ~0.8% để đạt mục tiêu 95%). Nguyên nhân là do ở kịch bản primary_timeout_100, luồng request đổ dồn toàn bộ sang provider Backup. Mà Backup bản thân có 5% tỉ lệ lỗi giả lập, dẫn tới một phần nhỏ request rơi vào Static Fallback (lỗi hệ thống). Để cải thiện, cần bổ sung thêm Backup Provider thứ 3 hoặc hạ thấp lỗi giả lập của backup.

Shared Redis Cache cho hệ thống Multi-Instance

Trong các hạ tầng production thực tế, ứng dụng Gateway thường chạy song song trên nhiều container/server (để load balancing). Bộ nhớ cache in-memory local sẽ dẫn tới việc dữ liệu không đồng nhất và lãng phí tài nguyên gọi API trùng lặp.

Vấn đề của In-Memory

Mỗi instance Gateway giữ cache riêng. Khi instance A lưu cache, instance B không biết và vẫn gọi API thật tốn tiền. Khi thêm mới instance (scale ngang), bộ cache trống trơn (cold start).

Redis giải quyết thế nào?

Tập trung toàn bộ cache vào một cụm Redis dùng chung. Mọi instance đều đọc ghi từ đây. Giúp tỉ lệ hit cache tối đa và các instance mới có sẵn dữ liệu ngay lập tức.

Bảo vệ dữ liệu & False Hit

Triển khai regex lọc từ khóa bảo mật và cơ chế phát hiện False Hit (nếu năm/số 4 chữ số khác nhau trong prompt tương tự thì bỏ qua) ngay trong luồng scan dữ liệu Redis.

Bằng chứng kiểm thử đồng bộ trạng thái (Shared State)

Chúng ta đã kiểm tra thành công sự đồng bộ dữ liệu giữa 2 instance cache song song trên cùng 1 server Redis:

Phân tích Điểm yếu còn lại & Khắc phục

Mặc dù hệ thống hiện tại đã đạt độ tin cậy rất cao và vượt qua toàn bộ các bài test, trong môi trường sản xuất thực tế (Production) vẫn tồn tại một số rủi ro cần lưu ý và giải pháp khắc phục tương ứng:

1. Rủi ro hiệu năng Redis khi cache phình to

Điểm yếu: Lệnh SCAN để tính toán similarity trực tiếp trên toàn bộ key của Redis sẽ bị chậm tuyến tính (O(N)) khi số lượng cache tăng lên hàng chục ngàn key. Điều này làm tăng CPU của Redis và tăng độ trễ.

Khắc phục: Thay thế cơ chế scan chuỗi bằng một **Vector Database** chuyên dụng (như Qdrant, Milvus) hoặc sử dụng tính năng Redisearch với vector embeddings để tìm kiếm khoảng cách cosine nhanh hơn ở độ phức tạp O(log N).

2. Lệch trạng thái Circuit Breaker

Điểm yếu: Trạng thái của Circuit Breaker hiện tại được lưu trữ in-memory cục bộ trên từng node Gateway. Nếu một node phát hiện Provider lỗi và ngắt mạch, các node khác vẫn tiếp tục gửi request lỗi sang Provider đó.

Khắc phục: Đồng bộ hóa trạng thái Circuit Breaker tập trung trên Redis bằng cách sử dụng các lệnh nguyên tử INCR và EXPIRE để đếm số lỗi đồng thời cho toàn bộ hệ thống.

Đề xuất Cải tiến Tiếp theo (Next Steps)

  1. Triển khai Rate Limiting tập trung: Bổ sung bộ giới hạn lưu lượng (Token Bucket) dựa trên IP hoặc API Key sử dụng Redis để bảo vệ các Provider phía sau khỏi tình trạng quá tải cục bộ.
  2. Graceful Degradation nâng cao: Đã được cấu hình dự phòng in-memory cache tự động khi Redis sập kết nối, đảm bảo cổng gateway luôn mở 100% thời gian hoạt động.
  3. Tối ưu hóa Chi phí thông minh (Cost-aware routing): Phát triển thuật toán tự động chuyển đổi luồng request sang mô hình rẻ hơn khi phát hiện ngân sách (budget) trong ngày sắp cạn kiệt.
